乾乾 (QianQian)：全球 Agentic 编程背景下的跨平台 AI 资产管理与语义协同演进深度研究报告第一部分：Agentic 编程时代的资产范式转移与主权回归在 2026 年的全球软件开发语境下，编程范式正经历着从“辅助增强”向“完全 Agentic（智能体化）”的根本性转变。新兴的 AI 智能体编程模式允许大语言模型（LLM）自主规划复杂的开发步骤，并直接与编译器、调试器、版本控制系统以及各种外部工具进行深度交互 。这种范式的跃迁使得开发者的核心资产不再仅仅是源代码，而是进化为一套包含提示词（Prompts）、原子技能（Skills）、角色定义（Agents）以及协作流（Flows）在内的“AI 能力资产” 。然而，这种能力的爆发式增长伴随着严重的“平台孤岛”挑战。当前的 AI 集成开发环境（IDE）如 Cursor、Claude Code 和 Open Code，各自建立了一套互不兼容的私有配置逻辑与资产定义标准，导致开发者的个人 AI 能力被锁定在特定的商业生态中 。这种平台锁入不仅阻碍了能力的跨环境复用，更导致了优质 Prompt 与 Skill 在对话结束后的流失 。在这一背景下，名为“乾乾 (QianQian)”的产品应运而生，其核心使命在于基于 GitHub 实现 AI 资产的主权化存储，并通过 CLI 调度实现能力的跨平台分发 。第二部分：主流 AI 编程平台的资产规范与配置层级深度解析为了实现资产的通用化管理，必须对当前最具代表性的 AI 开发平台进行底层解析。目前，Cursor、Claude Code 与 Open Code 代表了三种截然不同的设计哲学。2.1 Cursor：基于.mdc 的规则注入机制Cursor 作为目前最成熟的 AI IDE，其资产核心已从传统的 .cursorrules 演进为结构化的 .mdc（Markdown Configuration）文件 。每一个 .mdc 文件不仅是指令的载体，更是一个具备元数据感知能力的智能单元。核心属性行为逻辑与注入机制alwaysApply定义规则是否无条件注入系统提示词。设置为 true 时，规则将在所有对话会话中作为常驻上下文存在，而不论其相关性 。globs基于文件通配符的触发机制。当用户打开、修改或在对话中提及匹配特定模式的文件时，规则将被自动拉取并注入上下文 。description语义化激活的核心。AI 智能体会根据这一描述评估规则与当前任务的相关性。如果描述匹配，Agent 会调用 fetch_rules 工具拉取全文 。Cursor 的配置遵循极其严格的优先级合并逻辑：Team Rules（团队规则）优先级最高，随后是 Project Rules（项目规则），最后是 User Rules（用户规则）。这种分层设计确保了企业级规范（如安全策略）能够强制覆盖个人习惯，同时也为开发者留出了个性化微调的空间。2.2 Open Code：多模型编排与 7 层配置层级与 Cursor 的闭环生态不同，Open Code 强调去中心化与多模型协作。其架构设计采用了高度灵活的配置层级，支持从远程组织默认值到运行时指令的 7 层合并 。配置层级名称存储位置与作用域描述优先级评级Inline Config通过 OPENCODE_CONFIG_CONTENT 环境变量传递的临时覆盖 。7 (最高).opencode 目录存储于项目内部的 Agents、Commands、Plugins 及 Skills 。6Project Config项目根目录下的 opencode.json，可随 Git 提交 。5Custom ConfigOPENCODE_CONFIG 指定的路径，用于特定环境的临时覆盖 。4Global Config~/.config/opencode/opencode.json，存储用户全局偏好 。3Remote Config通过 .well-known/opencode 获取的组织级默认设置 。2Default Settings系统预置的内置模式（如 Build 模式和 Plan 模式）。1 (最低)Open Code 的独特之处在于其“纪律智能体”模型。例如，Sisyphus 智能体充当主编排器，负责规划任务并将其委派给不同的专业子智能体（如 Atlas 或 Oracle），这种层级化的编排架构极大地缓解了单一模型在处理长程任务时的上下文漂移问题 。2.3 Claude Code：基于安全域的作用域管理Claude Code 引入了更为现代的“作用域（Scope）”系统，其设计重点在于受管环境下的合规性与安全性 。Managed Scope: 位于系统级目录（如 Linux 下的 /etc/claude-code/managed-settings.json），专为 IT 管理员设计，用户完全无法覆盖，用于强制执行企业安全策略 。User Scope: 位于 ~/.claude/，跨项目生效，适合存储个人的 API 密钥及通用的编辑器主题偏好 。Project Scope: 存储在项目的 .claude/ 目录中，设计为随 Git 共享，用于在团队协作中统一 Tools、Permissions 和 Hooks 。Local Scope: 使用 .claude/*.local.json 文件，由 gitignore 保护，适用于开发者个人的项目级微调或测试 。Claude Code 的钩子（Hooks）系统支持 8 种以上核心事件，允许在 SessionStart、PreToolUse 或 PostToolUse 等关键节点触发自动化逻辑 。这种基于事件的扩展性是其实现复杂自动化工作流的基石。第三部分：“乾乾”资产定义规范：提示词、技能与智能体标准化“乾乾”产品的核心理念是将零散的能力固化为可版本管理的数字资产。为了实现这一目标，必须建立一套能够跨平台解析的自描述存储协议 。3.1 Prompts（提示词）的语义化层级在“乾乾”的定义中，提示词被划分为 System Prompts 和 Task Prompts。由于不同模型对指令的遵循能力存在差异，资产必须支持多格式表现 。研究发现，Claude 模型在处理基于 XML 标签的结构化提示词时表现尤为出色，而 GPT 模型则更倾向于基于原则（Principle-driven）的简洁指令 。格式类型适用场景与优势建议用途XML Tags复杂的长程任务，能够有效防止提示词注入并明确语义边界 。架构设计、大型重构Markdown兼顾人类可读性与模型解析，适合定义分层的指令体系 。代码评审、注释生成YAML适合定义参数化的提示词变量（如 framework, version） 。系统 Persona 定义3.2 Skills（技能）的执行边界与 manifest 规范技能（Skill）在“乾乾”中被定义为原子级的执行单元，通常包含 logic.py/js 逻辑以及一个声明式的 manifest.json 。参考 OpenClaw 的设计标准，技能必须明确其权限边界以确保系统安全 。JSON{
  "name": "performance-auditor",
  "permissions": {
    "filesystem": ["read"],
    "network": ["*.web.dev"],
    "shell": false
  },
  "dependencies": {
    "bins": ["lighthouse-cli"]
  }
}
这种 manifest 机制允许“乾乾”在同步资产时，自动检测本地环境是否满足 Skill 的运行条件（如特定的二进制工具是否在 PATH 中），并根据不同的 IDE 平台生成相应的适配配置 。3.3 Agents 与 Flows：角色定义与协作编排“乾乾”中的 Agents 定义高度参考了 OASF（Open Agentic Schema Framework）标准 。一个完整的智能体记录不仅包含指令（Instructions），还包括其具备的技能集（Skills）、知识库引用（Knowledge Base）以及预选的模型偏好 。Flows 则用于描述多 Agent 协作逻辑。目前的技术趋势倾向于将 Flow 定义为描述性的有向无环图（DAG），由目标执行引擎（如 Open Code 的编排器）解析并执行 。在“乾乾”的进阶阶段，需要引入轻量级的 Flow 执行引擎，支持基础的条件分支与状态持久化。第四部分：核心调度中心：基于 Git 的版本控制与同步算法资产主权是“乾乾”设计哲学的第一要务。通过 GitHub 私有仓库，开发者可以完全掌控自己的数字资产，而不依赖于任何特定的 AI 服务商 。4.1 Isomorphic-Git：轻量化存储引擎选型在实现 CLI 同步逻辑时，“乾乾”采用了 isomorphic-git 作为核心存储驱动。相比于 C 实现的 libgit2，纯 JavaScript 的 isomorphic-git 在 Node.js 环境下具备更优的安装便利性与跨平台一致性 。比较维度Isomorphic-Git (乾乾采用)Nodegit (原生绑定)环境依赖零原生依赖，纯 JS 。依赖特定平台的编译环境 。同步功能支持 Clone, Push, Pull 及 100% 互操作性 。功能全，但易产生段错误 (Segfault) 。资源消耗支持浅克隆 (Shallow clone)，节省流量 。只能进行完整克隆 。4.2 智能冲突处理：语义合并与 @sync 指令在多端同步场景下，冲突不可避免。“乾乾”通过在资产头部定义的 @sync 打标来决定同步冲突的解决策略。@sync:full: 强制以远程云端为准，本地修改将被覆盖。@sync:local: 忽略云端更新，保留本地针对特定项目的微调。@sync:merge: 调用 AI 介入的语义合并。AI 会自动梳理差异点，并在保持 Markdown 或 YAML 语法完整性的前提下，给出合并建议供用户确认 。第五部分：安全体系与沙箱执行：保护本地环境不受恶意 Skill 侵害随着 AI 技能开始具备执行 Shell 脚本和网络访问的能力，安全性成为了不可忽视的红线。OpenClaw 等平台的案例研究显示，恶意的 AI 技能可能会被设计用来窃取存储在 .env 文件中的 API 密钥或利用 OAuth 令牌进行越权操作 。5.1 三层隔离架构：微虚拟机、gVisor 与 Isolates为了实现 Skill 的安全执行，“乾乾”提供了分级的隔离方案 。隔离方案技术实现机制启动性能安全强度Isolates (v8)isolated-vm 在同一进程内创建独立的 JS 堆栈 。极快 (<10ms)低 (仅隔离 JS 堆栈) 。Deno Sandbox利用 Deno 原生的权限控制模型执行代码 。快 (<50ms)中 (受限于内核权限) 。Firecracker硬件辅助的微虚拟机，具备独立的 Guest 内核 。中 (~150ms)高 (硬件级隔离) 。对于普通 Prompt 的应用，“乾乾”采用默认的无隔离模式；但对于包含逻辑执行的 logic.py/js 技能，系统会自动建议开启基于 Deno 或微虚拟机的沙箱环境，以防止生成的恶意代码逃逸到宿主机系统 。5.2 敏感信息处理：个人配置文件的非对称管理为了防止敏感信息（如 OpenAI API Key）被意外提交到 GitHub，“乾乾”采用了类似于 Open Code 的分层存储逻辑。全局敏感配置存储于本地 ~/.acl/credentials.yaml 中，并使用 AES-256-GCM 加密 。资产库中仅保留 .env.template 文件，实际运行时的环境变量由 CLI 在部署到特定 IDE 平台时动态注入。第六部分：AI 驱动的维护入口：反向总结与一键入库“乾乾”不仅是资产的搬运工，更是资产的打磨者。AI 管理入口负责将零散的开发过程固化为持久化的能力 。4.4.2 Auto-Doc：从对话到资产的自动化流程当开发者在 Cursor 或 Open Code 中完成了一段卓有成效的对话时，通过 acl capture --ai 命令，AI 会自动扫描对话历史 。特征提取: AI 识别出对话中被证明有效的 Prompt 模式或逻辑链条 。规范化: AI 自动生成符合 MDC 规范的 Frontmatter，并编写 Markdown 说明文档 。提交建议: AI 生成清晰的 Git Commit Message，并建议将其归类到 /prompts/tasks/ 目录下。
这种流程确保了开发者的最佳实践能够随着每一次对话被持续沉淀，而不是随对话框关闭而消失 。第七部分：兼容性矩阵与多平台分发机制“乾乾”的核心价值在于其“主动分发机制”。CLI 会主动向各个平台注入配置，而非被动等待平台调用 。3.3.2 兼容性分级标准系统通过 compatibility.yaml 动态维护资产与各大平台的兼容性状态 。Full (完全兼容): 资产功能在目标平台 100% 可用（如纯 Markdown 规则同步到 Cursor）。None (不兼容): 核心依赖缺失（如需要本地 Docker 的运维 Agent 同步到纯云端的 Cloud Code）。
这种分级机制解决了“平台无关性”带来的功能落差问题。当用户尝试同步一个不兼容的资产时，CLI 会发出清晰的警告并提供降级建议（例如：将 Skill 降级为仅供参考的 Prompt 提示词） 。第八部分：结论与未来展望：从能力库到“互联网认知”中枢“乾乾 (QianQian)”的诞生标志着 AI 编程工具链从“功能导向”向“资产导向”的重大转型。通过将 AI 能力资产化，开发者不仅获得了摆脱单一商业平台锁定的自由，更建立了一套可持续累积、可跨团队分发的数字大脑 。展望未来，“乾乾”将进一步集成 A2A（Agent2Agent）协议，实现跨平台智能体的无缝协作 。随着 MCP 2.0 标准的普及，个人 AI 资产库将成为开发者在“互联网认知”时代的核心竞争力。每一次“终日乾乾”的沉淀，都将成为构建下一个伟大产品的智能基石。总结建议行动方案根据目前的需求澄清结果，建议开发团队在 MVP 阶段重点攻克以下模块：P0 级: 完成基于 isomorphic-git 的核心同步逻辑，并深度适配 Cursor 的 .mdc 文件格式。P0 级: 实现基础的 @sync 冲突处理引擎，支持 full 和 local 策略。P1 级: 引入基于 Deno 的轻量级沙箱，用于安全执行 Skill 逻辑。P1 级: 开发 acl capture --ai 功能，实现从 IDE 交互到资产库的一键入库流程。通过这套渐进增强的开发路径，“乾乾”将迅速成长为全球开发者必备的跨平台 AI 资产管理利器。
