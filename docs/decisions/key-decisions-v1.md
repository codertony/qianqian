## 🔴 P0 决策（本周内必须确认）

### 决策 1: MVP 功能边界

**问题**: Phase 1 的 MVP 是否需要支持 Skill 执行？

**背景**:
- 当前类型定义完整，但执行引擎缺失
- Skill 执行需要沙箱设计，增加复杂度
- everything-claude-code 中 Skill 主要是 Prompt 集合

**选项**:

| 选项 | 描述 | 影响 | 推荐度 |
|------|------|------|--------|
| **A** | **仅作为 Prompt 集合（推荐）** | - 简化 MVP，减少 2-3 周工作量<br>- 快速验证核心同步流程<br>- Phase 2 再考虑执行 | ⭐⭐⭐⭐⭐ |
| B | 基础 Node.js 执行 | - 增加执行风险<br>- 需要权限控制设计<br>- 复杂度适中 | ⭐⭐⭐ |
| C | 完整沙箱执行 | - 增加 3-4 周工作量<br>- 需要 Deno/vm2 调研<br>- 安全性高 | ⭐ |

**我的建议**: 选 **A**
- 符合"渐进增强"原则
- 参考 everything-claude-code 的实践
- 快速交付 MVP 验证价值

**决策确认**: ☐ 选项A  ☐ 选项B  ☐ 选项C

### 决策 2: CLI 命令优先级

**问题**: 8 个 CLI 命令（sync/pull/push/status/list/show/diff/capture）的实现顺序？

**背景**:
- 当前仅 `init` 完成，其余都是 TODO
- 需要优先打通核心用户流程
- 时间资源有限（Phase 1 剩余 ~55h）

**推荐优先级**:

```
P0（立即实现）:
├── sync      # 核心同步流程，打通价值链条
├── pull      # 从云端获取更新
├── status    # 查看状态，用户高频使用
└── capture   # 捕获资产，核心功能

P1（Phase 1 后期）:
├── push      # 推送到云端
└── list      # 列出资产

P2（可延后到 Phase 2）:
├── show      # 查看详情
└── diff      # 查看差异
```

**我的建议**: 优先实现 P0 的 4 个命令
- sync：打通 Cursor ↔ GitHub ↔ OpenCode 完整流程
- pull/status：用户日常使用最高频
- capture：验证资产捕获核心能力

**决策确认**: ☐ 同意推荐优先级  ☐ 需要调整（请说明）

### 决策 3: 平台支持范围

**问题**: Phase 1 支持哪些平台？

**背景**:
- Cursor 和 OpenCode 适配器类型定义已完成
- Claude Code 和 Cloud Code 需要额外调研
- 平台越多，测试矩阵越复杂

**选项**:

| 选项 | 平台 | 工作量 | 风险 | 推荐度 |
|------|------|--------|------|--------|
| **A** | **Cursor + OpenCode（推荐）** | +0 周 | 低 | ⭐⭐⭐⭐⭐ |
| B | A + Claude Code | +1 周 | 中（文档待调研） | ⭐⭐⭐ |
| C | B + Cloud Code | +2 周 | 高（文档极少） | ⭐ |

**调研参考**:
- Cursor: .mdc 格式明确，四种激活模式清晰
- OpenCode: 7 层配置，YAML 格式，文档完善
- Claude Code: 5 层作用域，需进一步调研
- Cloud Code: 与 OpenCode 差异不明，文档稀缺

**我的建议**: 选 **A**
- 覆盖 80% 用户场景
- 降低 Phase 1 风险

### 决策 4: 适配器工厂实现方式

**问题**: `createAdapter()` 工厂如何实现？

**背景**:
- 当前返回 null，完全未实现
- 需要支持动态适配器实例化
- 考虑未来扩展性

**选项**:

| 选项 | 实现方式 | 优点 | 缺点 | 推荐度 |
|------|---------|------|------|--------|
| **A** | **硬编码 Map（推荐）** | - 简单可靠<br>- 类型安全<br>- 易于调试 | - 新增平台需改代码 | ⭐⭐⭐⭐⭐ |

**我的建议**: 选 **A**
- MVP 阶段简单优先
- 后续可通过依赖注入扩展
- 符合"先跑起来，再优化"原则

**决策确认**: ☐ 选项A  ☐ 选项B  ☐ 选项C

## 🟡 P1 决策（Phase 1 内确认）

### 决策 5: 配置层级设计

**问题**: 如何处理 Open Code 的 7 层配置 vs 乾乾的 3 层配置？

**背景**:
- 乾乾设计: Defaults → User → Project → Env → CLI（5层）
- Open Code: Remote → Global → Project → Local → Env → Inline → Managed（7层）
- 需要平衡简单性和兼容性

**选项**:

| 选项 | 设计 | 说明 | 推荐度 |
|------|------|------|--------|
| **A** | **保持乾乾 5 层（推荐）** | - 简化用户理解<br>- 通过 sync_policy 实现类似效果<br>- 不与平台强耦合 | ⭐⭐⭐⭐⭐ |
| B | 融合 7 层 | - 完全兼容 Open Code<br>- 配置更强大 | ⭐⭐⭐ |
| C | 可配置层级 | - 用户自定义<br>- 灵活但复杂 | ⭐ |

**我的建议**: 选 **A**
- 保持产品独立性
- 5 层已覆盖 95% 场景
- 映射到平台时做转换即可

**决策确认**: ☐ 选项A  ☐ 选项B  ☐ 选项C

## 🟢 P2 决策（Phase 2-3 确认）

### 决策 8: Flow 执行引擎范围

**问题**: Flow 是仅作为描述性配置，还是支持执行引擎？

**背景**:
- Flow 类型定义已完成（nodes/edges/variables）
- 执行引擎需要调度、状态管理、错误处理
- 目标平台（OpenCode）可能有自己的执行引擎

**选项**:

| 选项 | 范围 | 影响 | 推荐度 |
|------|------|------|--------|
| **A** | **仅描述性配置（推荐）** | - Flow 作为资产同步<br>- 由目标平台执行<br>- 减少复杂度 | ⭐⭐⭐⭐⭐ |
| B | 轻量级执行引擎 | - 基础节点调度<br>- 简单的条件分支<br>- 适中复杂度 | ⭐⭐⭐ |
| C | 完整执行引擎 | - 并行执行、状态持久化<br>- 类似 Airflow<br>- 高复杂度 | ⭐ |

**参考**: 
- everything-claude-code 中 Flow 是描述性的
- oh-my-opencode 有 Sisyphus 编排系统

**我的建议**: 选 **A**
- 符合"协议先行"原则
- 避免重复造轮子
- 目标平台执行更有优势

**决策确认**: ☐ 选项A  ☐ 选项B  ☐ 选项C
